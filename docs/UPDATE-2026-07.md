### A-1. モバイル UI 改善(カード被り・アクション視認性)

問題: `PokerTable` は aspect-[2/1] 絶対配置。狭い画面で `SeatView`(カード2枚+スタック)が
巨大になり座席同士とボードが重なる。アクションパネルはテーブル下でスクロールが必要。

対応:
1. `SeatView` — CPU 席(非ヒーロー・非ショーダウン)の裏向きカード2枚を **モバイルでは非表示**
   (`hidden sm:flex` をカード行に適用)。代わりに何も出さない(ポジション+スタックのみの
   小さなピルになる)。padding も `p-1.5 sm:p-2`、min-width は `min-w-[56px] sm:min-w-[80px]`
   程度に調整。フォールド済み席も同様に縮む。
2. ヒーロー席のカードはモバイルでも表示を維持する(自分のハンドは常に見える)。
3. ヒーローのアクションパネル(「あなたの番」の Panel)を **モバイルで sticky 化**:
   ラッパーに `sticky bottom-20 z-30 md:static md:bottom-auto`(モバイル下部タブバー
   fixed bottom-0 の上に載せる)+ Panel に `bg-surface/95 backdrop-blur` 相当を付け、
   スクロール位置に関係なくアクションが見えるようにする。
4. `BetControls` のボタンはタッチ向けに `py-3`(モバイル)を確保。プリセット5個は
   `flex-wrap` 可。
5. `PotDisplay`・`BoardView` はセンター配置のまま。ボードカードは `size="sm"` を
   モバイル、`md` 以上で現状サイズ(現状の実装を確認しつつ、重なる場合のみ調整)。

### A-3. セッション再開

方針: ハンド間の `SessionState`(core/game/session.ts)を localStorage に永続化し、
未終了セッションを「次のハンドから」再開できるようにする。ハンド途中の盤面は保存しない。

`src/store/sessions.ts` 拡張(persist version を 2 に上げ、migrate で旧データ維持):
```ts
type ActiveSession = { recordId: string; state: SessionState; savedAt: number };
// state に追加:
activeSession: ActiveSession | null;
saveActiveSession: (recordId: string, state: SessionState) => void; // 同期・local のみ
clearActiveSession: () => void;
```

`useVersusSession` 変更:
- `start()`: `createSession` の id 解決後に `saveActiveSession(id, session)`
- ハンド確定 effect(`commitHandResult` 後): `saveActiveSession(recordId, nextSession)`。
  セッション終了時(`!canContinue`)と `quit()` では `clearActiveSession()`
- 新 API `resume(saved: ActiveSession)`: session/sessionId を復元し
  `startHand(null, configForHand(state), state.seatStacks)` で次ハンド開始。
  buttonSeat は復元不要(null 開始で 0 から)、簡易仕様として許容し UI に明記不要。

`SessionTab` UI: 未開始画面(`!started`)の先頭に、`activeSession` が存在し
`status === 'active'` かつ `canContinue(state)` の場合のみパネルを表示:
「進行中のセッションがあります — {形式} / {ハンド数}ハンド / スタック {n}」
+ [再開する] ボタン(resume して started=true)+ [破棄] ボタン(clearActiveSession)。
再開時の注意書き: 「ハンド途中の状態は保存されないため、次のハンドから再開します」

### A-4. 単発モードのチップ引き継ぎ

1. `useVersusGame.newHand()`: 現在は `startHand(prev, config)` でスタックが毎回
   `startingStack` にリセットされる。`prev.players` の最終スタックを
   `seatStacks` 引数として渡し引き継ぐ。ただし **0 以下の席は startingStack(100) に
   自動リバイ**(単発モードにバスト概念はない)。ヒーローがリバイした場合、
   結果パネルか状態行に「スタックを100bbに補充しました」の一言を出す(実装容易な範囲で)。
2. タブ切替でのリセット防止: `Versus.tsx` で `GameTab` / `SessionTab` / `HistoryTab` を
   条件レンダーではなく **常時マウント + `hidden` クラスで出し分け** に変更する
   (`<div className={cn(tab !== 'game' && 'hidden')}>` 方式)。単発/セッション
   トグルも同様。これにより対戦ページ内の移動では状態が保持され、
   他ページへの遷移(アンマウント)でのみリセットされる(要求仕様どおり)。

## WP-B: コール判断ドリル改善 + インプライドオッズ + レンジ訓練UI

対象ファイル: `src/pages/PotOdds.tsx`, `src/core/potOdds.ts`, `src/core/potOdds.test.ts`,
`src/pages/RangeTrainer.tsx`

### B-1. ポット表記の明確化

前提(現行ロジックの規約・変更しない):
- ドロー判断・必要勝率タブ: `potOdds(pot, toCall) = toCall/(pot+toCall)` —
  表示中の「ポット」は **相手のベット額を含んだ額**。
- MDF タブ: `mdf(pot, bet) = pot/(pot+bet)` — こちらの「ポット」は
  **相手のベット前の額**(相手は bet をリスクして pot を取りに来る)。

対応(UI テキストのみ):
- ドロー判断・必要勝率タブ: ラベルを「ポット(相手のベット込み)」にし、
  問題カード下に注記「※ ポットには相手のベット額がすでに含まれています」。
- MDF タブ: ラベル「ポット(相手のベット前)」+ 注記「※ 相手はこのポットに対して
  ベットしています(ポットにベット額は含まれません)」。

### B-2. 新規タブ: インプライドオッズ判断

`PotOdds.tsx` に4つ目のタブ `implied`(ラベル「インプライドオッズ」)を追加。

core 追加 (`src/core/potOdds.ts`):
```ts
/** コールを損益分岐にするために将来ストリートで追加回収が必要な額。
 *  equity*(pot+toCall+X) = toCall を X について解く。負なら 0。
 *  X = toCall/equity - (pot + toCall) */
export function requiredImpliedAmount(pot: number, toCall: number, equity: number): number;

export type ImpliedDrill = {
  pot: number;          // 相手のベット込み
  toCall: number;
  outs: number;
  drawId: string;
  drawLabel: string;
  street: 'flop' | 'turn';
  behindStack: number;  // 相手の残りスタック(有効スタック)
  collectFactor: number;
  requiredExtra: number; // requiredImpliedAmount の結果
  answer: 'call' | 'fold';
};

export function genImpliedDrill(rng?: () => number): ImpliedDrill;
```

生成仕様:
- draw は `DRAW_TYPES` から選択。street は flop/turn。equity は
  `equityFromOuts(outs, street==='flop' ? 2 : 1)`。
  ※ フロップの2枚読みは「ターンでの追加ベットに再度直面しない」前提の簡易化。
- pot: 40..300(10刻み)、toCall: pot の 0.4〜1.2 倍を 10 刻み丸め(最低 20)。
- **直接オッズでは足りないケースを主とする**: 生成後に
  `equity >= potOdds(pot,toCall)` なら toCall を増やして再計算(最大数回)。
  それでも直接コール可なら、そのまま採用してよい(answer は下の式で決まる)。
- collectFactor(将来回収の期待係数、簡易モデル):
  - フラッシュ系 (`flush`, `fd-gutshot`, `fd-oesd`): 0.3(完成が読まれやすい)
  - ストレート系 (`gutshot`, `oesd`): 0.5(隠れやすい)
  - `two-overcards`: 0.2(完成しても回収困難)
- behindStack: 100..600(10刻み)。
- 判定: `answer = requiredExtra <= collectFactor * behindStack ? 'call' : 'fold'`
  (requiredExtra が 0、つまり直接オッズ十分でも 'call')

UI(DrawView を踏襲):
- 問題カード: ポット(ベット込み) / 相手のベット / 相手の残りスタック /
  ストリート+ドロー+アウツ。
- 回答: フォールド / コール の2択。
- フィードバックに表示: 必要勝率、実勝率、不足を埋める必要追加回収額
  `requiredExtra`、見込み回収額 `collectFactor * behindStack`(「このドローは
  {読まれやすい/隠れやすい}ため回収期待は残りスタックの約{n}%」の説明文)、
  正解理由の一文。
- 注記: 「これは簡易モデルによる目安であり、厳密なソルバー解ではありません」。
- サイドバー「考え方」: 追加回収必要額 = コール額 ÷ 実勝率 −(ポット+コール額)、
  隠れたドローほど回収期待が大きい、等。
- 進捗記録: `recordPotOdds` を再利用。attempts は
  `drillKind: 'potOdds'`, `scenarioId: 'implied:pot=..,call=..,outs=..,behind=..'`
  (attempts の DrillKind union は変更しない)。

テスト(`potOdds.test.ts` に追加):
- `requiredImpliedAmount`: equity ちょうどで 0、代表値の手計算一致、負にならない。
- `genImpliedDrill`: シード付き rng で invariants(範囲、answer と式の整合)を検証。

## WP-C: クイズ 50 問化

- `QUIZ_QUESTIONS` を **50 問以上**に拡張。既存 10 問は原則維持(明らかな矛盾があれば修正可)。
- 構成の目安:
  - プリフロップ open/fold 判断(context 付き、fold/raise 2択): 25 問前後。
    各ポジション(UTG/HJ/CO/BTN/SB)にレンジ境界付近のハンドを散らす。

  - ポットオッズ/必要勝率/MDF の暗算問題(4択、数値はきれいな値): 8 問前後。
    解説に計算式を必ず書く。
  - アウツ/2&4の法則: 4 問前後。
  - 用語・基本戦略(`src/data/glossary.ts` の記述と矛盾しないこと): 5 問前後。
- **整合性が最重要**: プリフロップ問題の正解はアプリ内レンジデータ
  (`src/core/ranges/` — yokosawa.ts が ground truth、変更禁止)と一致させること。
- 新規テスト `quizQuestions.test.ts`:
  1. id がユニーク。
  2. answer が choices のいずれかの value に含まれる。
  3. `context.heroPos && context.hand` を持ち choices が fold/raise の全問について、
     アプリのレンジ API(`getRfiScenarios` / `getEffectiveRange` + `primaryAction` 等。
     RangeTrainer.tsx の使い方を参照)から得た期待アクションと answer が一致する。
     mode はまず 'tournament' で既存 10 問が通るか確認し、通る mode を採用して
     テストに固定・コメントで明記する。モードによって答えが変わるハンドは
     問題文に前提を書くか採用しない。
  4. 問題数 >= 50。

## WP-D: 新機能「相手目線レンジ」トレーナー

対象ファイル: 新規 `src/pages/PerceivedRange.tsx`, `src/App.tsx`,
`src/components/AppShell.tsx`, `src/pages/Home.tsx`, `src/store/progress.ts`,
`src/store/attempts.ts`, `src/pages/Review.tsx`(ラベル追加のみ)

目的: 「自分のアクションによって相手から自分のハンドレンジがどう見えているか」を
意識する訓練。既存のプリフロップレンジデータを利用(新データ不要)。

- ルート `/perceived`、ページタイトル「相手目線レンジ」、
  説明「自分のアクションが相手にどんなレンジを見せているかを意識する訓練。」
- ナビ: `AppShell.tsx` の「学習(ドリル)」セクション(クイズの下あたり)に
  icon `Eye`(lucide)で追加。MOBILE_NAV は 5 枠固定のため変更しない。
  `Home.tsx` の機能カード一覧にも同様のカードを1枚追加(既存カードの体裁に合わせる)。
- タブ2つ:

### D-1. タブ「レンジ内？」
- シナリオ: `getRfiScenarios('tournament')`(6-max RFI)からランダム。
- ランダムなハンドクラス(169 クラスから一様、または deck から2枚引いて
  `cardsToHandClass`)を提示し、
  「あなたは {scenario.label}。相手から見て、この {hand} は
  あなたが見せているオープンレンジに含まれる?」
- 2択: 含まれる / 含まれない。期待値: `primaryAction(range[hand]) === 'raise'`。
- 回答後: `RangeGrid`(`highlight={hand}`)で「相手から見えているあなたのレンジ」を
  表示し、「あなたの実際のハンドが何であれ、相手にはこのレンジ全体に見えている」
  という趣旨の解説文を出す。

### D-2. タブ「レンジ%当て」
- シナリオ同上。「あなたは {pos} からオープンした。相手から見たあなたのレンジは
  全スターティングハンドの約何%?」
- 正解: `openPercent(range)`(core/ranges/expand)。選択肢4つ:
  正解(%表示は小数0桁)と、×0.5 / ×1.5 / ×2.2 を丸めた誤答(重複時はずらす)。
- 回答後: RangeGrid + 「タイトなポジションほど見せるレンジが狭い=強く見える」
  等の解説。
